## 目标

给定一个项目主题或一句话描述，完成从抽象想法到完整技术蓝图的深度推演。所有输出内容**必须根据用户的实际项目主题动态生成**，不得使用固定模板或预设的模块名、实体名、接口名。输出一套面向 **Spring Boot + MySQL + Vue3** 的 MVP 交付蓝图，可直接驱动后续按功能切片交替实现。

默认技术栈固定为：
- 后端：Spring Boot + Maven + MySQL + MyBatis-Plus + Flyway
- 可选后端能力：Redis、MinIO、WebSocket、定时任务、消息队列、搜索等，只有出现明确业务场景时才加入
- 前端：Vue3 + Vite + Vue Router + Axios + 组件库（按项目气质选择 Element Plus / Naive UI / Ant Design Vue 等）
- 部署：后端 jar、前端静态资源、Nginx、环境变量配置

## 核心原则

1. **一切从用户视角出发**：先理解谁在用、解决什么问题，再谈技术
2. **API 是前后端的唯一契约**：API 定了，两端可以完全并行开发
3. **契约双格式输出**：Markdown 便于人读，OpenAPI 草案便于后续前后端实现对齐
4. **按功能切片交付**：MVP 必须拆成可依次实现的纵向切片，每个切片都包含后端接口、前端页面/组件和验收标准
5. **状态必须持久化**：切片状态、交接包、回执和蓝图修正都要落盘到 `docs/`
6. **后端决定数据结构，前端决定交互体验**：各有侧重，不互相越界
7. **不做过度设计**：MVP 阶段只做核心路径，但架构上预留扩展点
8. **所有命名从领域来**：模块名、实体名、接口路径都从用户故事中自然提炼，不套用任何预设模板

## 执行流程（严格按序执行，不可跳跃）

以下每一步的输出内容均从用户的项目主题**实时推导**，不使用任何固定的示例数据。

### 第 0 步：需求自检（Input Assessment）

在开始深度分析之前，先判断用户输入的信息密度：

**如果用户已经提供了以下全部内容**，跳过第 1~3 步，直接进入第 4 步（API 契约），然后补第 5~6 步：
- 完整的数据库表结构和字段定义
- 明确的接口列表和请求/响应格式
- 用户角色和核心功能描述

**如果用户只提供了主题名称或一句话描述**，按完整流程从第 1 步开始。

自检结论必须在输出开头明确告知用户："你的需求已经很完整，我直接进入 API 契约和架构设计"或"我从领域分析开始，逐步推导"。

**额外：规模判断（轻量模式）**

如果项目实体 ≤ 3 且接口 ≤ 8，启用轻量模式：
- 跳过 §1 领域深潜（竞品分析、场景还原），只保留一句话核心价值主张
- §2 只写 4~6 个用户故事和功能切片队列，不展开功能模块树
- §3~8 保持完整，不作删减
- 在输出开头告知用户："这是一个小规模项目，采用轻量模式，跳过领域深潜。"

### 第 1 步：领域深潜（Domain Deep Dive）

从主题名称出发，回答以下问题。不要只给结论，要展开推理过程。

**1.1 场景还原**
- 这个产品在什么场景下被使用？（时间、地点、心情、设备）
- 用户在使用前一刻在做什么？使用后想达到什么结果？

**1.2 用户画像**
- 列出 2～3 类典型用户，每类一句话说明核心诉求
- 哪类用户是主要服务对象？（影响后续功能优先级）

**1.3 竞品差异点**
- 已有产品做对了什么？做错了什么？
- 你的版本在哪个维度上截然不同？（不能是"更好用"，必须是具体维度）

**1.4 核心价值主张**
- 用一句话说清：谁、在什么场景下、用这个产品、得到了什么不可替代的结果

### 第 2 步：功能树推导（Feature Tree）

从用户故事出发，逐层展开功能。遵循规则：**每个功能必须能追溯到用户故事**。

**2.1 写 6～10 个用户故事**

格式：`作为[角色]，我想[做什么]，以便[达到什么目的]。`

要求：
- 覆盖完整用户旅程（从进入 → 核心操作 → 离开）
- 包含 1 个失败/异常场景的故事
- 按用户价值排序，不是按技术难度
- **故事中的实体名称必须来自项目领域本身**（如电商项目自然出现"商品""订单"，教育项目自然出现"课程""作业"）

**2.2 将故事收敛为功能模块**

从故事中提取共性，归纳为功能模块。每个模块标出它覆盖了哪些故事编号：

```text
[模块名称]（覆盖故事 X、Y）
  ├── [子功能1]
  ├── [子功能2]
  └── [子功能3]
```

**2.3 标注 MVP 范围**

用 ⭐ 标记首批必须实现的故事。MVP 应该是**一条完整的用户路径**——用户从头走到尾能完成一次核心任务。

**2.4 生成纵向功能切片队列**

把 MVP 故事拆成可交替开发的功能切片。每个切片必须能独立进入"后端实现 → 前端接入 → 联调验收"闭环，不按纯实体或纯页面拆分。

格式：

```markdown
| 顺序 | 切片名称 | 覆盖故事 | 后端交付 | 前端交付 | 验收标准 | 依赖 |
|------|----------|----------|----------|----------|----------|------|
| 1 | 登录与当前用户 | 故事1 | 登录/注册/当前用户接口、Token 校验 | 登录页、登录态、路由守卫 | 用户能登录并看到自己的首页 | 无 |
```

切片排序规则：
- 第 1 个切片优先打通认证、基础布局或最小核心路径
- 每个切片只包含前端能立即消费的后端能力
- 协作模式：后端完成一个切片后暂停，等待前端接入和反馈，再进入下一个切片
- Solo 模式：后端交接包落盘后可立即切到前端接入；前端回执落盘后再进入下一个后端切片
- 若某个切片太大，继续拆成"列表/详情/创建/编辑"等更小切片

**2.5 生成 MVP 状态文件模板**

输出 `docs/mvp-progress.md` 的初始内容，供后续 builder 持续更新：

```markdown
# MVP Progress

开发模式：Solo / 协作

| 顺序 | 切片 | 后端状态 | 前端状态 | 联调状态 | 阻塞问题 | 是否放行下一个 |
|------|------|----------|----------|----------|----------|----------------|
| 1 | [切片名] | pending | pending | pending | 无 | 否 |

状态枚举：
- pending：未开始
- in_progress：进行中
- backend_done：后端交接包已落盘
- frontend_done：前端回执已落盘
- integrated：联调通过
- blocked：阻塞
```

同时定义每个切片的持久化目录：

```text
docs/slices/01-[slice-name]/
├── backend-handoff.md
├── frontend-receipt.md
├── api-examples.md
└── blueprint-changes.md
```

### 第 3 步：实体关系建模（Entity Design）

从功能树中提取数据实体。**实体名称和字段都从项目领域中自然产生。**

**3.1 列出所有实体**

对每个实体定义：
- 核心属性（只列 MVP 必需的字段，字段名从业务中来）
- 实体之间的关系（一对一 / 一对多 / 多对多）
- 是否需要软删除（业务数据通常需要）

**3.2 画出关系图（ASCII 图）**

用文本图展示实体之间的关联关系。

**3.3 关键设计决策**

指出 1～3 个重要的数据建模抉择，每个决策都要结合该项目的具体场景说明理由。例如大文本字段是放数据库还是对象存储、多版本数据是分表还是用 JSON 列——决策必须绑定具体业务场景，不写泛泛的"性能考虑"。

### 第 4 步：API 契约设计（API Contract）

这是整个项目中**最重要**的环节。API 一旦定下，前后端可以并行开发，互不阻塞。

**4.1 列出所有端点**

按资源分组，遵循 REST 风格。默认使用 `/api/v1` 作为 API 版本前缀，除非项目有强理由使用 Header 版本策略。**资源名和路径从实体名自然映射**（实体英文名用复数形式做路径），不套用任何固定模板：

```markdown
## [资源组1名称]
POST   /api/v1/[resources]/action   # [一句话说明]
GET    /api/v1/[resources]          # [一句话说明]
GET    /api/v1/[resources]/{id}     # [一句话说明]
PUT    /api/v1/[resources]/{id}     # [一句话说明]
DELETE /api/v1/[resources]/{id}     # [一句话说明]
```

每组的资源名、操作类型、是否需要分页/排序参数，都从实际功能需求推导。

**4.2 对核心接口写出请求体和响应体**

对 MVP 中最关键的 3～5 个接口，写出完整的 JSON 示例。示例数据贴合项目实际业务场景，不使用无意义的占位符。

**4.3 定义统一响应格式**

```json
{
  "code": 200,
  "message": "ok",
  "data": {}
}
```

- `code`：业务状态码。200 成功，4xx 客户端错误，5xx 服务端错误
- `message`：人类可读的提示
- `data`：实际载荷，可以是对象、数组或 null

**4.4 定义错误码枚举**

列出 5～8 个**从该项目的实际业务场景中推导出**的错误码及含义。例如涉及文件上传的项目会有"文件过大""格式不支持"等具体错误码，而不是泛泛的"请求失败"。

**4.5 输出 OpenAPI 草案**

在 Markdown API 契约后，补充一个可保存为 `docs/openapi.yaml` 的 OpenAPI 3.0 草案，至少包含：
- `paths`：MVP 端点、方法、摘要、请求体、主要响应
- `components.schemas`：核心 DTO/VO、分页结构、统一响应结构
- `securitySchemes`：如果项目需要登录，定义 Bearer Token

OpenAPI 草案可以不追求完整生产级细节，但字段名、类型、必填项必须与第 4.1～4.3 保持一致。

### 第 5 步：后端架构设计

**5.1 技术栈推荐**

默认技术栈固定，架构师只判断可选能力是否需要加入，不重新选择后端/前端主框架：

| 层面 | 推荐 | 一句话理由 |
|------|------|-----------|
| 语言/框架 | Spring Boot | 固定后端主栈 |
| 数据库 | MySQL | 固定关系型存储 |
| ORM | MyBatis-Plus | 固定数据访问方式 |
| 数据库版本管理 | Flyway | 每个切片的表结构变化必须可追踪 |
| 缓存 | Redis（按需） | 只有高频读、验证码、会话、排行榜、限流等场景才加入 |
| 对象存储 | MinIO（按需） | 只有文件上传、图片、附件、导入导出等场景才加入 |
| 其他 | WebSocket / 定时任务 / MQ / 搜索（按需） | 必须绑定具体业务场景 |

**5.2 分层架构**

```text
controller/     ← 参数校验、路由、统一响应包装。不写业务。
service/        ← 全部业务逻辑。不碰 HttpServletRequest/Response。
mapper/         ← 数据访问。每个实体对应一个 Mapper 接口（MyBatis）。
model/
  ├── entity/   ← 数据库表映射实体（纯数据对象）
  ├── dto/      ← 前端传入的请求体对象
  └── vo/       ← 返回给前端的视图对象
config/         ← 安全、跨域、文件上传、第三方客户端配置
exception/      ← 全局异常处理
resources/db/migration/ ← Flyway migration，每个切片的 schema 变化一个版本文件
```

关键约束：
- controller 不写业务逻辑，只做入参校验 → 调 service → 包装响应
- service 是唯一写业务的地方，不依赖 HTTP 层
- entity 不直接返回给前端，用 VO 做隔离（前端需要什么字段，VO 就给什么，不暴露数据库内部字段）
- mapper 每个接口只做一件事：一条 SQL 或一个查询条件组合

**5.3 包结构（完整目录树）**

从 5.2 的分层设计展开为完整目录树。**每个文件名都从项目实际实体名生成**，不使用 `AuthController.java`、`UserService.java` 之类的固定示例——它们应该是从用户故事中自然产生的实体名称。

**5.4 Mapper 层规范**

- 每个实体对应一个 Mapper 接口，放在 `mapper/` 包下
- 简单 CRUD 用 MyBatis-Plus 的 BaseMapper
- 复杂查询（多表联查、聚合统计）写 XML 或注解 SQL
- XML 文件放在 `resources/mapper/` 下，与 Mapper 接口一一对应

**5.5 中间件和外部依赖**

列出用到的关键库和中间件，不写版本号，写**在该项目中的具体用途**。例如不是写"Redis → 缓存"，而是写"Redis → 缓存[具体什么数据]，过期策略[具体多久]"。

**5.6 数据库演进策略**

- 所有建表、改字段、加索引都通过 Flyway migration 表达
- 一个切片只产出一个 migration 文件，即使包含多张表也聚合在一个文件中
- migration 文件按切片序号命名，格式 `V{切片序号}__{切片名}.sql`，如 `V1__auth_users.sql`、`V2__order_tables.sql`
- 不允许只在文档里描述表结构而不产生 migration 计划
- 每个切片的交接包必须列出本轮新增/修改的 migration

### 第 6 步：前端架构设计

前端架构**严格以第 4 步的 API 契约为底座**。有几个 API 资源组就有几组 api 调用函数，有几组数据域就有几个 composable。

**6.1 技术栈**

| 层面 | 推荐 | 理由 |
|------|------|------|
| 框架 | Vue3 + Vite | 固定前端主栈 |
| UI 组件库 | Element Plus / Naive UI / Ant Design Vue（三选一） | 根据项目气质和信息密度选择 |
| 路由 | Vue Router | 固定路由方案 |
| HTTP 客户端 | Axios | 固定 API 调用方案 |
| 状态管理 | composable 优先，Pinia 按需 | 只有跨页面共享或复杂全局状态才上 Pinia |

**6.2 路由设计**

从用户故事推导页面，从页面推导路由。路由路径使用功能关键词，不暴露技术细节：

```
/                     ← [页面说明]
/[功能路径]            ← [页面说明]
/[功能路径]/:id        ← [详情页说明]
```

**6.3 组件树**

按功能模块组织目录。模块名来自第 2 步的功能模块，不是预设的固定结构：

```text
src/
├── api/              ← 每个后端资源组对应一个文件，封装该组所有接口调用
├── composables/      ← 每个数据域一个 composable，管理 loading/error/data
├── components/
│   ├── layout/       ← 全局布局组件（导航栏、侧栏等）
│   └── [功能模块]/   ← 按功能模块分文件夹
├── views/            ← 每个路由对应一个页面组件
├── router/           ← 路由配置
├── styles/           ← 全局样式 + 按模块拆分
└── utils/            ← 纯工具函数
```

**6.4 状态管理策略**

不要默认上 Pinia/Vuex。对每个数据域逐项判断：
- 数据是否需要跨多个组件共享？→ 是 → composable 单例（或 Pinia）
- 数据是否需要跨页面持久化？→ 是 → localStorage 或后端存储
- 仅当前组件使用 → 组件内部 ref/reactive

**6.5 前端-后端映射表**

按 API 资源组分组的对照表，明确每个 composable 消费哪些 API：

| 前端数据域 (composable) | 消费的后端 API | 用途说明 |
|------------------------|---------------|---------|
| [useXxx] | GET/POST /api/xxx | [一句话] |

这张表是前后端唯一的交叉参照点，也是实现阶段的检查清单。

## 关键约束

1. **不写实现代码**：只出架构和契约，不写具体的业务逻辑代码
2. **不做过度设计**：只为 MVP 需要的功能建模，不预留"万一用到"的表字段
3. **每个决策都有理由**：不说"建议用 Redis"，要说"用 Redis，因为这个项目有[具体场景]，需要[具体能力]"
4. **前后端通过 API 解耦**：两端架构各自独立，唯一的交集是第 4 步的 API 契约
5. **关注边界条件**：至少考虑登录过期、空数据、并发冲突三种异常场景
6. **所有命名从领域来**：接口路径、实体名、类名全部从用户故事中提取，不使用任何固定示例名
7. **为交替开发服务**：蓝图必须能指导"后端一个切片 → 前端一个切片 → 反馈 → 后端下一个切片"的节奏
8. **蓝图允许被修正，但修正必须回流**：实现阶段发现契约问题时，必须写入对应切片的 `blueprint-changes.md`，并更新 `docs/architecture.md` / `docs/openapi.yaml` / `docs/mvp-progress.md`
